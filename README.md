# Vulnerable Flask Lab & Automated RCE Exploit

## 📝 Описание проекта
Данный пет-проект представляет собой легковесную тренировочную лабораторию (мишень), написанную на языке **Python** с использованием микрофреймворка **Flask**. В проект входит уязвимое веб-приложение, содержащее критические баги из списка OWASP Top 10, а также кастомный скрипт-эксплойт для автоматизации атаки.

Лаборатория разработана с учетом жестких ограничений по ресурсам системы (развертывание занимает < 50 МБ и использует базу данных SQLite прямо в оперативной памяти `:memory:`).

## 🛡️ Реализованные уязвимости

1. **SQL Injection (Insecure String Formatting)**
   * **Где искать:** Эндпоинт `/search?name=`
   * **Суть:** Из-за прямой склейки SQL-запроса хакер может обойти логику аутентификации с помощью пэйлоада `admin' OR '1'='1` и слить всю БД.
   
2. **Blind SQL Injection (Boolean-based)**
   * **Где искать:** Эндпоинт `/blind-search?name=`
   * **Суть:** Сервер не выводит данные на экран, а отвечает бинарно (Да/Нет). Уязвимость позволяет вытаскивать секреты посимвольно с помощью функций `SUBSTR()`.

3. **Insecure Direct Object Reference (IDOR)**
   * **Где искать:** Эндпоинт `/profile?user_id=`
   * **Суть:** Отсутствие проверки прав доступа (Access Control). Изменяя ID в параметре, можно читать конфиденциальные данные других пользователей.

4. **Reflected XSS с обходом фильтрации (WAF Bypass)**
   * **Где искать:** Эндпоинт `/secure-greet?name=`
   * **Суть:** Разработчик попытался блокировать теги `<script>`, но фильтр обходится использованием альтернативных HTML-обработчиков событий (например, `<img src=x onerror=... >`).

5. **Local File Inclusion (LFI) & Log Poisoning**
   * **Где искать:** Эндпоинт `/advanced-lfi?file=`
   * **Суть:** Приложение позволяет читать файлы сервера. Так как входящий заголовок `User-Agent` логируется в текстовый файл, это открывает вектор атаки для внедрения вредоносного кода в логи и перехода к удаленному выполнению команд (RCE).

## 💻 Скрипт автоматизации (Exploit)

В репозитории также находится файл `ex.py` — автоматический эксплойт на Python для цепочки атак **LFI ➡️ SSTI ➡️ RCE**.

### Как работает эксплойт:
1. Используя библиотеку `requests`, скрипт отправляет HTTP-запрос, подменяя заголовок `User-Agent` на вредоносную конструкцию шаблонизатора Jinja2: `{{ self.__init__.__globals__['__builtins__']['__import__']('os').popen('whoami').read() }}`.
2. Сервер логирует этот «яд» в файл `server_access.log`.
3. Скрипт обращается к лог-файлу через уязвимость LFI. Функция `render_template_string` на сервере выполняет код, и скрипт перехватывает вывод системной команды (например, имя пользователя `kali`).

## 🛡️ Эволюция уязвимостей в проекте: от базовых к продвинутым

В ходе разработки проект был модернизирован от простых "учебных" багов до продвинутых сценариев, имитирующих реальные обходы систем защиты (WAF/Фильтрации):

### 1. SQL Injection & Blind SQLi
* **Базовый уровень:** Прямая склейка строк, позволяющая легко вытащить данные через `admin' OR '1'='1`.
* **Продвинутый уровень (Реализовано):** **Blind SQL Injection (Boolean-based)**. Сервер скрывает ошибки и результаты запросов, выдавая лишь статус "Найден/Не найден". Реализован вектор атаки для посимвольного перебора конфиденциальных данных из БД с использованием SQL-функций манипуляции строками.

### 2. XSS & WAF Bypass
* **Базовый уровень:** Обычная инъекция `<script>`, отрабатывающая в браузере.
* **Продвинутый уровень (Реализовано):** **Фильтрация тегов (WAF Simulation)**. В код добавлен бэкэнд-фильтр, который вырезает опасные теги `<script>` и `</script>`. Реализован обход фильтрации (Bypass) через альтернативные HTML-контексты и обработчики событий (например, использование конструкции `<img src=x onerror=... >`), что доказывает неэффективность "черных списков" в веб-защите.

### 3. LFI & Цепочка атак до RCE (Log Poisoning)
* **Базовый уровень:** Простой Path Traversal для чтений файлов конфигурации типа `win.ini` или `passwd`.
* **Продвинутый уровень (Реализовано):** **Атака класса Log Poisoning + Template Injection**. Проект симулирует ведение системных логов сервера. С помощью подмены заголовков (`User-Agent`) в утилиту логирования доставляется вредоносный payload. При последующем чтении логов через LFI, движок Flask (`render_template_string`) принудительно исполняет внедренный Python-код, что переводит уязвимость из разряда "чтения файлов" в **полный захват контроля над сервером (Remote Code Execution)**.
